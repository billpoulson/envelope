Certificates & sealed secrets

Use this when clients must encrypt sensitive values before they reach Envelope. The server stores ciphertext and per-recipient wrapped data keys; it does not hold private keys and cannot decrypt payloads by design.

Security chain & flow

Public material and ciphertext live in Envelope; plaintext exists only on clients that perform encryption or hold a recipient private key.

Role of certificates

Each registered certificate is a recipient identity: an X.509 certificate whose public key (typically RSA) is used to wrap a symmetric data key. Whoever holds the matching private key can unwrap that key offline and decrypt the payload.

Envelope stores:

  • A short display name you choose (unique among certificates).
  • The full PEM text of the certificate.
  • A SHA-256 fingerprint (hex) derived from the parsed certificate bytes — used to detect duplicates and to help you confirm you registered the right file.

Envelope does not store private keys, PKCS#12 bundles, or passwords. Upload public certificate material only (for example .crt / PEM from your PKI or openssl req / openssl x509 output).

Registering a certificate (UI)

  1. Open Admin → Certificates in the header.
  2. Enter a name you will recognize in lists and API payloads (e.g. ci-west-2025).
  3. Paste the PEM block, including -----BEGIN CERTIFICATE----- / -----END CERTIFICATE----- lines.
  4. Submit. If the PEM is invalid, the app returns a validation error. If the same name or the same fingerprint already exists, registration is rejected (409 via API).

Registering a certificate (API)

Admin scope required for all certificate routes.

  • GET /api/v1/certificates — list certificates (id, name, fingerprint, created_at).
  • POST /api/v1/certificates — JSON body {"name": "…", "certificate_pem": "…"}. Returns 201 with the created row.
  • DELETE /api/v1/certificates/{id} — remove a certificate. Prefer the web UI for deletion: the UI blocks removal if any sealed secret still references that certificate, so you do not strand metadata.

If you delete via API while recipients still reference the certificate, the database may reject the delete depending on integrity rules—remove or update sealed secrets first.

Deleting certificates

In the web UI, delete is blocked while a certificate is still referenced by any sealed secret recipient row. Remove the sealed secret (or replace it with a new upload that drops that recipient) first, then delete the certificate.

Rotating a recipient usually means: register the new cert, upload a new sealed secret row with wrapped keys for the new cert (and other recipients as needed), then remove the old cert once nothing references it.

Sealed secrets on a bundle

Each bundle can have zero or more sealed secret rows, keyed by key_name (like an environment variable name). For each row, Envelope stores:

  • Algorithm identifiers and base64 ciphertext, nonce, and optional additional authenticated data (AAD) for the payload.
  • For each recipient: certificate_id (matching a registered certificate), wrapped_key (base64), and key_wrap_alg (default rsa-oaep-256).

Read and write follow normal bundle API key scopes: GET/POST/DELETE /api/v1/bundles/{name}/sealed-secrets with bundle read/write as appropriate. In the UI, use the bundle’s Sealed secrets sub-nav.

Typical client-side flow

  1. Register every recipient’s public certificate in Envelope (admin); note each certificate_id from the API or UI.
  2. Generate a random data key and encrypt your plaintext with AEAD (for example AES-256-GCM), producing ciphertext, nonce, and optional AAD.
  3. For each recipient, wrap the data key with that certificate’s public key (RSA-OAEP with SHA-256 is the default key_wrap_alg in the API).
  4. POST one JSON body per key_name (upsert): payload fields plus a recipients array with certificate_id, wrapped_key, and key_wrap_alg.
  5. Consumers with a private key fetch metadata via the API, unwrap the data key, and decrypt the payload locally. Envelope never performs that decryption.

Example request body

{
  "key_name": "API_TOKEN",
  "enc_alg": "aes-256-gcm",
  "payload_ciphertext": "BASE64_CIPHERTEXT",
  "payload_nonce": "BASE64_NONCE",
  "payload_aad": "optional-context",
  "recipients": [
    {"certificate_id": 1, "wrapped_key": "BASE64_WRAPPED_KEY_1", "key_wrap_alg": "rsa-oaep-256"},
    {"certificate_id": 2, "wrapped_key": "BASE64_WRAPPED_KEY_2", "key_wrap_alg": "rsa-oaep-256"}
  ]
}

Duplicate certificate_id entries in one request are deduplicated. Every certificate_id must exist at post time or the API returns 400 with unknown ids.

Security reminders

  • Treat ciphertext, wrapped keys, and export responses as sensitive: anyone with recipient private material can recover plaintext.
  • Standard bundle/stack export and opaque env URLs deliver Fernet-decrypted variables for automation—they do not include sealed-secret payloads.
  • Use HTTPS in production; never log PEM bodies, wrapped keys, or API keys.
